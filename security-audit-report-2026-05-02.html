Security & Runtime Audit

Xiaozhi ESP32 Server

Audit nhanh repo, Docker runtime, logs, cấu hình deploy và hành động shutdown theo yêu cầu ngày 2026-05-02.

Repo: /root/xiaozhi-esp32-server
Runtime compose: /root/dockercompose/docker-compose.yml
Kết luận: Không thấy bằng chứng hack, nhưng service đang lỗi nghiêm trọng
Action: Backend Xiaozhi đã shutdown

1. Executive Summary

Kết luận điều tra

Chưa tìm thấy dấu hiệu repo bị cài backdoor, reverse shell, miner, cron lạ, SSH key lạ hoặc tiến trình runtime đáng ngờ. Rủi ro chính hiện tại là cấu hình deploy yếu và backend đang rơi vào restart loop.

Sự cố runtime chính

xiaozhi-esp32-server có RestartCount=6775. Log cho thấy server gọi http://127.0.0.1:8002/xiaozhi/config/server-base và nhận 502 Bad Gateway, sau đó crash/restart liên tục.

Trạng thái shutdown

Đã tắt auto-restart và stop các container backend: server, MySQL, Redis, phpMyAdmin. Container web/nginx vẫn giữ lại vì nó đang làm reverse proxy chung cho nhiều domain khác, không chỉ Xiaozhi.

Đánh giá: Đây giống lỗi cấu hình/deployment gây self-DoS hơn là một vụ hack đã được xác nhận. Tuy vậy, vì có secret trong config runtime, port public và sandbox bị nới lỏng, nên cần hardening trước khi bật lại.

3. Runtime Incident Detail

Observed failure chain

xiaozhi-esp32-server starts
  -> reads /opt/xiaozhi-esp32-server/data/.config.yaml
  -> manager-api.url = http://127.0.0.1:8002/xiaozhi
  -> POST /xiaozhi/config/server-base
  -> nginx on :8002 proxies /xiaozhi/ to http://127.0.0.1:8003
  -> port 8003 belongs to the Python server that is still starting / crashing
  -> 502 Bad Gateway
  -> config loader raises HTTPStatusError
  -> container exits and restarts

This is a circular dependency/deploy wiring problem: the server depends on a config endpoint that currently routes back to the server-side OTA/API port instead of a healthy manager API service.

4. Findings

SeverityFindingWhereImpactFix
Critical Backend server crash loop, 6775 restarts. Docker state/logs Service unavailable, log noise, resource churn. Fix manager-api.url and ensure manager API is a real upstream, then re-enable restart policy.
High Secrets and operational tokens exist in runtime config/compose. /root/dockercompose/docker-compose.yml, /root/dockercompose/data/.config.yaml Credential exposure risk if host/repo/logs are leaked. Rotate secrets, move to Docker secrets or env files outside repo, restrict file permissions.
High Public ports on shared nginx edge include 80, 443, 8002. xiaozhi-esp32-server-web Admin/API surface is internet-reachable if Cloudflare/firewall allows it. Bind admin port to localhost/VPN, add auth/rate limit, block direct public 8002.
Medium seccomp:unconfined disables Docker syscall sandboxing. main/xiaozhi-server/docker-compose*.yml, runtime compose Container escape blast radius is larger if an app-level exploit lands. Remove unless a verified model/audio dependency requires it; prefer targeted capabilities.
Medium Repo sample compose contains default passwords such as 123456. main/xiaozhi-server/docker-compose_all.yml, application-dev.yml Developers may deploy sample defaults by accident. Replace with placeholders and add .env.example.
Medium Setup script uses third-party GitHub acceleration proxy. docker-setup.sh Supply-chain trust includes the proxy endpoint. Use direct GitHub release URLs or pin checksums.
Medium server.auth.enabled: false in base config sample. /root/dockercompose/data/config.yaml If effective config falls back to base config, devices can bypass token checks. Verify effective config from runtime and enable auth before public use.

6. Recommended Recovery Before Re-enable

  1. Rotate MySQL, manager API and any provider/API tokens found in runtime config.
  2. Move secrets out of compose/YAML into Docker secrets or root-only env files.
  3. Fix manager-api.url so Python server talks to the real manager API upstream, not an nginx route that loops to its own OTA port.
  4. Run manager API as a visible container/process and verify /xiaozhi/config/server-base returns 200 before starting Python server.
  5. Remove seccomp:unconfined and drop unnecessary privileges.
  6. Bind admin port 8002 to localhost/VPN or protect it with edge allowlist and auth.
  7. Enable device auth for production and remove sample allowed devices.
  8. After remediation, re-enable restart only for healthy services: docker update --restart=always ....

7. Confidence & Limits

  • This audit was read-only for repo/content and runtime inspection, except for the requested shutdown action.
  • The Python server image lacks ps, so process inspection inside that container was limited; crash-loop logs and Docker state were available.
  • No disk forensics, packet capture, host auth log analysis or image-layer diffing was performed in this pass.
  • Current evidence supports: service misconfiguration and unhealthy deployment, not a confirmed external compromise.
